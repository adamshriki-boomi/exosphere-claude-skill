# Error Messages

#### Purpose

Textual design element to help you fix a problem in a task

#### Function

Error messages alert users to a problem that already happened. Explains what happened, why it happened, and how to fix the problem, where applicable.

#### General Use

-   For alerting users when info in a form field is not formatted or entered correctly and needs to be corrected
-   For when the system is having trouble through no fault of the user, or when the website is down, or user data is missing
-   For reducing repeat errors and mistakes
-   When users need to fix their mistakes when inputting or submitting their info

#### Details

-   Errors messages focus on how the user can fix the problem.
-   Use \[problem\], then \[solution\] structure in most cases. For example: "To delete an Integration Pack, provide a valid pack ID."
-   Error messages are written in clear, simple language
-   Messages are written in sentence case with no point of view (POV), which means neutral language to avoid blaming the user. \[POV section\]
-   Don't use negative words like Sorry. Use positive language to inform the user about the error like a coach would and what they need to do to fix it.
-   Do use Sorry sparingly if something is failing because of a system or performance issue on our end, particularly after a form submission, or fetching data.
-   Do not use words like Please to avoid increasing or lessening severity of message. Only use when user is asked to do something inconvenient like waiting for system to reload, etc., and rarely.
-   Don't use technical terms or jargon that will confuse the user, even if the users are peer users. These terms do not always localize or translate well.
-   Use sentence case to avoid all caps.
-   Do not use words like Try again more than once to avoid loops. If needed, be sure to include follow up message with specific next steps, such as Contact us or Email us for help.
-   Do not use exclamation points or bold lettering.
